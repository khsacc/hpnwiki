## Commands

```bash
yarn dev       # Generate _meta.json + start TinaCMS dev server + Next.js (localhost:3000)
yarn build     # Generate _meta.json + tinacms build + next build
yarn start     # Serve production build
```

Local dev requires no external services ‚Äî `MONGO_URI` absent triggers local-filesystem mode automatically.

Admin UI: `http://localhost:3000/admin/index.html`

## Architecture

### Rendering layer (Nextra v2, Pages Router)

- `pages/index.mdx` ‚Äî site root
- `pages/docs/**/*.mdx` ‚Äî all wiki content, served at `/docs/*`
- `pages/_app.tsx` ‚Äî global KaTeX CSS import
- `theme.config.tsx` ‚Äî Nextra DocsTheme: logo, sidebar, TOC, and the "TinaCMS „ÅßÁ∑®ÈõÜ" edit button (renders only for `pages/docs/` files)

### Sidebar navigation (auto-generated)

`pages/**/_meta.json` files are **never hand-edited and are gitignored**. They are generated by `scripts/generate-meta.mjs` at the start of every `dev` and `build`. The script reads two frontmatter fields from each `.mdx` file:

- `title` ‚Äî sidebar label
- `sidebarOrder` ‚Äî sort order (ascending; defaults to 999)

Subdirectory labels and order come from that directory's `index.mdx`. Always set `title` and `sidebarOrder` when creating a new page.

### CMS layer (TinaCMS self-hosted, no TinaCloud)

- `tina/config.ts` ‚Äî schema with two collections: `TinaUserCollection` (auth) and `doc` (all wiki pages at `pages/docs/`)
- `tina/database.ts` ‚Äî **must use `export default`** (TinaCMS CLI calls `require(file).default`). Switches between `createLocalDatabase()` and MongoDB-backed `createDatabase()` based on whether `MONGO_URI` is set.
- `pages/api/tina/[...routes].ts` ‚Äî Next.js API handler wrapping `TinaNodeBackend`; also the production auth handler (Auth.js / `next-auth` v4)
- `netlify/functions/tina.ts` ‚Äî standalone Netlify Function alternative (not used when `@netlify/plugin-nextjs` converts API routes automatically)

**Local vs production mode** is governed by a single flag evaluated in both `tina/config.ts` and `tina/database.ts`:

```ts
const isLocal = process.env.TINA_PUBLIC_IS_LOCAL === 'true' || !process.env.MONGO_URI
```

In local mode: no `authProvider`, no `contentApiUrlOverride`, no MongoDB.  
In production mode: `UsernamePasswordAuthJSProvider`, `contentApiUrlOverride: '/api/tina/gql'`, MongoDB via `mongodb-level`.

### Deployment (Netlify)

`netlify.toml` uses `@netlify/plugin-nextjs`, which converts `pages/api/` routes to Netlify Functions automatically. The build command is `yarn build` (which includes the meta generation step).

## Environment variables

See `.env.example`. Key distinction:

| Variable | Local dev | Production |
|---|---|---|
| `MONGO_URI` | unset (triggers local mode) | required |
| `TINA_PUBLIC_IS_LOCAL` | `true` or unset | `false` |
| `NEXTAUTH_SECRET` | optional | required |
| `GITHUB_PERSONAL_ACCESS_TOKEN` | not needed | required (Contents: read/write) |

## Content conventions

- All wiki pages live in `pages/docs/` as `.mdx` files
- Nested sections are subdirectories; each subdirectory must have an `index.mdx`
- `content/docs/` is unused (legacy path from initial setup ‚Äî ignore it)
- `content/users/index.json` holds the users array; password field must be TinaCMS PBKDF2 format: `{ "value": "<salt_hex><key_hex>", "passwordChangeRequired": false }` ‚Äî salt=32 bytes, key=512 bytes, iterations=25000, digest=sha256. bcrypt strings do NOT work.
- `__ref__/` contains reference material (old HTML wiki) ‚Äî not served by Next.js
